## Step 1: Method Selection

### Quick Selector — What Do You Need to Know?

| Research Question | Best Method | Min. Participants | Time Needed |
|-------------------|-------------|-------------------|-------------|
| "Can users complete this task?" | Moderated Usability Test | 5 per segment | 1-2 weeks |
| "Where do users get confused?" | Unmoderated Test (Maze/Lookback) | 15-20 | 3-5 days |
| "Do they understand this in 5 seconds?" | 5-Second Test | 20+ | 1-2 days |
| "Which design do they prefer?" | Preference Test / A/B | 20+ per variant | 3-5 days |
| "Is the navigation findable?" | Tree Test | 20-30 | 3-5 days |
| "Does the IA make sense?" | Card Sorting (Open/Closed) | 15-20 | 3-5 days |
| "Quick feedback, low budget?" | Guerrilla Testing | 5-8 | 1 day |
| "Live product performance?" | Analytics + Session Recording | All users | Ongoing |
| "Overall usability score?" | SUS Survey | 10+ | 1-2 days |

### Context-Based Method Guide

**Student project / learning:**
- Guerrilla testing (5 users, coffee shop, 20 min each)
- OR Unmoderated with free Maze tier
- Time: 1-2 days total

**Client project / professional:**
- Moderated usability test (5-8 users per persona)
- + SUS survey for benchmark score
- Time: 1-2 weeks

**Time crunch (need feedback in 24h):**
- 5-second test for first impressions
- OR rapid guerrilla (3-5 users, 15 min each)

**Remote users (Indian Tier 2/3):**
- WhatsApp video call + screen share
- Hindi/regional language facilitator
- Unmoderated with simplified tasks

## Step 3: Participant Screener

Match EXACTLY to primary persona from define-ideate. Wrong participants = wrong data.

```
SCREENER QUESTIONS — [Product Name]

QUALIFYING QUESTIONS (must answer correctly to participate):
Q1: [Behavior qualifier — e.g., "How often do you order food online?"]
    ✅ Qualify: [answer range — e.g., "1+ times per week"]
    ❌ Disqualify: ["Never" or "Rarely"]

Q2: [Device qualifier — e.g., "Which phone do you use?"]
    ✅ Qualify: [Android mid-range]
    ❌ Disqualify: [iPhone — different mental model for some flows]

Q3: [Experience qualifier — e.g., "Have you used UPI in the last month?"]
    ✅ Qualify: [Yes]
    ❌ Disqualify: [No]

DISQUALIFYING QUESTIONS (immediately exclude):
✅ Works in UX/design/tech industry (expert bias)
✅ Has participated in our research before (familiarity bias)
✅ [Any other bias source specific to your product]

DIVERSITY REQUIREMENTS:
✅ Mix: [% Tier 1 / % Tier 2 / % Tier 3 users]
✅ Age range: [min-max]
✅ Gender: [balanced]
✅ Language: [Hindi / English / both]

SAMPLE SCREENER MESSAGE (for WhatsApp/social outreach):
"Hi! I'm a UX design student working on a [product type] app.
I'm looking for people who [qualifying behavior] to try my
app for 20 minutes. It's free, [with/without] ₹[X] gift card.
Interested? Reply here."
```

## Step 4: Test Script

### Moderator Introduction (Read Verbatim)

```
INTRODUCTION SCRIPT:

"Namaste / Hi [Name]! Thank you for your time today.

My name is [your name]. I'm testing a design I've been
working on — I want to understand how real people use it.

Three important things before we start:

1. I'm testing the DESIGN, not you. There are no right or
   wrong answers. If something is confusing, it's my fault —
   not yours. That's exactly what I need to learn.

2. Please think out loud as you go. Say whatever you're
   thinking — "I'm looking for...", "I'm confused because...",
   "I expected this to..."

3. I may not answer questions during the tasks — I want to
   see what you do naturally. After each task I'll explain.

Do you have any questions before we start?

[Start recording — with permission]
'Is it okay if I record this session for my notes only?
No one else will see it.'"
```

### Warm-Up Questions (3-5 minutes)

```
WARM-UP:
"Before I show you the app, tell me a bit about yourself."

• "How do you currently [relevant behavior]?"
• "What apps do you use for [related task]?"
• "What do you like/dislike about them?"

[Goal: Understand mental model before they see your design]
```

### Task Scenarios (Core of the Test)

```
TASK FORMAT — Always scenario-based, never feature-based:

✅ GOOD (scenario): "Imagine you want to [real situation].
                    Please show me how you would do that."

❌ BAD (feature): "Click on the payment button and complete
                   the transaction." [Tells them what to do]

TASK 1 — [Most critical flow]
Scenario: "[Real situation that would trigger this flow]"
Success criteria: [User reaches X screen / completes Y action]
Time limit: [X minutes]
Observation focus: [What to watch for]

TASK 2 — [Second critical flow]
Scenario: "..."
Success criteria: [...]
Time limit: [...]

TASK 3 — [Edge case / error recovery]
Scenario: "[Situation that triggers an error or edge case]"
Success criteria: [User recovers without help]
Time limit: [...]

TASK 4 — [Discovery / navigation task]
Scenario: "[Task that requires finding something in IA]"
Success criteria: [Found in X taps / X seconds]

POST-TASK QUESTIONS (after each task):
• "What were you thinking when [specific moment]?"
• "Was anything confusing or unexpected?"
• "What did you expect to happen when you [action]?"
• "On a scale of 1-10, how easy was that task?"
```

### Post-Test Questions (5 minutes)

```
WRAP-UP QUESTIONS:

Overall impression:
• "What's your overall impression of this app?"
• "What was the most confusing part?"
• "What did you like most?"
• "What one thing would you change first?"

SUS Survey (standardized score):
[Give 10-question SUS form — see references/sus-scale.md]

Final open question:
• "Is there anything important I didn't ask that you
   think I should know?"
```

During the test, record observations using this structure:

```
OBSERVATION LOG — [Participant ID]: P[1-8]
Persona match: [Yes/Partial/No]
Session date: [date]
Moderator: [name]

TASK OBSERVATIONS:
┌────────┬──────────┬────────────────┬─────────────┬────────────┐
│ Task   │ Success? │ Time taken     │ Errors made │ Quotes/    │
│        │ Y/N/Part │ vs expected    │             │ Observations│
├────────┼──────────┼────────────────┼─────────────┼────────────┤
│ Task 1 │          │                │             │            │
│ Task 2 │          │                │             │            │
│ Task 3 │          │                │             │            │
│ Task 4 │          │                │             │            │
└────────┴──────────┴────────────────┴─────────────┴────────────┘

BEHAVIOR CODES (use while observing):
S+ = Smooth, no friction
S- = Struggled, hesitated
E  = Error / wrong path
Q  = Asked a question (note it)
" " = Direct quote (note exact words)
💡 = Insight / unexpected behavior
⚠️ = Assumption violated

PARTICIPANT QUOTES TO CAPTURE:
• Exact words when confused: "..."
• Exact words when delighted: "..."
• Exact words about expectations: "..."
```

## Step 6: Analysis Framework

### Rainbow Sheet (For 5+ Participants)

```
RAINBOW ANALYSIS SHEET — [Product Name]

FINDING              │ P1 │ P2 │ P3 │ P4 │ P5 │ COUNT │ SEVERITY
═══════════════════════════════════════════════════════════════
[Issue 1]            │ 👎 │ 👎 │    │ 👎 │    │  3/5  │ MAJOR
[Issue 2]            │ 👎 │    │ 👎 │    │ 👎 │  3/5  │ MAJOR
[Issue 3]            │ 👎 │    │    │    │    │  1/5  │ MINOR
[Positive finding 1] │ 👍 │ 👍 │ 👍 │    │ 👍 │  4/5  │ KEEP

RULE: If 3+ out of 5 users have same issue → it's real, fix it.
```

### Severity Rating System

```
SEVERITY 4 — CRITICAL (fix before launch)
• User cannot complete primary task
• Data loss / money loss risk
• User gives up and leaves

SEVERITY 3 — MAJOR (fix in next sprint)
• User completes task but with significant struggle
• 3+ users hit same issue
• Task time 2x+ expected

SEVERITY 2 — MINOR (schedule for future)
• Small confusion, user recovers independently
• 1-2 users affected
• Cosmetic but noticed

SEVERITY 1 — POLISH (nice to have)
• Preference, not problem
• "I wish..." not "I can't..."
• Single user opinion
```

### SUS Score Interpretation

```
SUS SCORE BENCHMARKS:
90-100  — A+ Exceptional (top 10% of products)
80-89   — A  Excellent (above average)
70-79   — B  Good (above average — acceptable to ship)
68      —    Industry average (exactly average)
60-67   — C  OK (below average — needs improvement)
50-59   — D  Poor (major usability problems)
<50     — F  Unacceptable (do not ship)

INDIAN MARKET ADJUSTMENT:
• First-time users / Tier 2/3 — Target ≥ 75 (more forgiving threshold)
• Experienced users / Tier 1 — Target ≥ 80
• Enterprise / Complex tools — ≥ 70 acceptable
```

### Affinity Mapping (For Qualitative Findings)

```
STEP 1: Write every observation/quote on separate sticky note
STEP 2: Group silently — no talking while grouping
STEP 3: Name each group with a verb phrase ("Users can't find X")
STEP 4: Rank groups by frequency + severity
STEP 5: Top 3 groups = top 3 design priorities

DIGITAL TOOLS:
• FigJam (free, Figma integrated)
• Miro (free tier available)
• Google Jamboard (free)
• Even Google Slides works for student projects
```

## Anti-Patterns — NEVER DO THESE

```
❌ Testing with colleagues or family
   → They know too much / won't challenge you
   → Not your real users = not real data

❌ Leading questions during test
   ❌ "Don't you think the button is obvious?"
   ❌ "Did you notice the X feature?"
   ✅ Ask open questions only

❌ Explaining the design during test
   ❌ "Oh that button means..." = invalidates the test
   ✅ Stay silent — note the confusion instead

❌ Testing too late (only on Hi-Fi)
   ✅ Test Lo-Fi wireframes too — cheaper to fix early
   ✅ "Test early, test often" = less rework

❌ Ignoring positive findings
   ✅ What users loved = what NOT to change in iteration
   ✅ Senior designers protect what works

❌ Testing wrong participants
   ✅ 5 perfect participants > 20 wrong ones
   ✅ Wrong screener = misleading data

❌ Reporting findings without fixes
   ❌ "Users were confused by X" = meaningless
   ✅ Always pair: finding + evidence + specific fix

❌ Skipping SUS survey
   ✅ 10 questions, 2 minutes
   ✅ Gives you a benchmark score to track over time

❌ Treating all feedback as equal
   ✅ Severity 4 ≠ Severity 1
   ✅ Prioritize ruthlessly — you can't fix everything

❌ One round of testing = done
   ✅ Test → Fix → Test again
   ✅ At least 2 rounds for any serious product
```

## Quick Testing Guide — By Situation

### Situation 1: Student Project (Low Budget, Fast)
```
Method:      Guerrilla testing
Participants: 5 real users (not friends/family)
Where:        College campus, library, chai shop
Time:         20 min per session
Tools:        Phone camera + Google Form for SUS
Cost:         ₹0-500 (chai for participants)
Output:       5 observation sheets + SUS scores + iteration brief
```

### Situation 2: Client Project (Professional)
```
Method:      Moderated usability test
Participants: 5-8 per persona segment
Where:        Video call (Zoom/Google Meet) or in-person
Time:         45-60 min per session
Tools:        Figma prototype + Loom/Zoom recording + Maze
Cost:         ₹500-2000 per participant incentive
Output:       Full report + rainbow sheet + iteration brief
```

### Situation 3: Quick Validation (24 Hours)
```
Method:      5-second test + preference test
Participants: 20+ (use Maze free tier or social media)
Where:        Online / WhatsApp survey
Time:         5 min per participant
Tools:        Maze / UsabilityHub / Google Form
Cost:         ₹0
Output:       First impression data + preference winner
```

### Situation 4: Live Product
```
Method:      Analytics + session recording + micro-survey
Participants: All users (quantitative)
Where:        In-product
Time:         Ongoing
Tools:        Hotjar (free tier) / Microsoft Clarity (free) / Mixpanel
Cost:         ₹0-X
Output:       Drop-off analysis + heatmaps + NPS trend
```
